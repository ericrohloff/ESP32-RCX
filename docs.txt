LEGO RCX IR UART Protocol Documentation
========================================
Documented from working ESP32 implementation, April 2026


OVERVIEW
--------
The LEGO RCX communicates over IR using a standard UART serial protocol
modulated onto a 38kHz infrared carrier wave. It is NOT a standard IR
protocol like NEC or RC5. It is raw UART transmitted optically.


PHYSICAL LAYER
--------------
Carrier Frequency : 38kHz
Carrier Duty Cycle: 33% (values between 17-33% work)
Baud Rate         : 2400 baud
Bit Duration      : ~417 microseconds (1,000,000 / 2400)

IR Light ON  = Logic LOW  (0) - UART SPACE
IR Light OFF = Logic HIGH (1) - UART MARK

The line idles with the IR light OFF (logic HIGH).


BYTE FRAMING
------------
Each byte is transmitted as 12 bits in this order:

  Bit 1   : START BIT       - always light ON  (logic LOW)
  Bits 2-9: DATA BITS        - LSB first (least significant bit first)
  Bit 10  : PARITY BIT       - odd parity (see below)
  Bit 11  : STOP BIT         - always light OFF (logic HIGH)
  Bit 12  : INTER-BYTE GAP   - always light OFF (logic HIGH, idle)

Total: 12 bits per byte at 417us each = ~5ms per byte


DATA BIT ENCODING
-----------------
  Data bit 0 -> UART LOW  -> IR light ON
  Data bit 1 -> UART HIGH -> IR light OFF


PARITY BIT
----------
Odd parity is used. The parity bit is chosen so that the total number
of 1-bits across the 8 data bits plus the parity bit is always ODD.

  Count the number of 1-bits in the byte:
    - If count is already odd  -> parity bit = 0 (light ON)
    - If count is even         -> parity bit = 1 (light OFF)

Example:
  0x55 = 0b01010101 -> four 1-bits (even) -> parity bit = 1 (light OFF)
  0xFF = 0b11111111 -> eight 1-bits (even) -> parity bit = 1 (light OFF)
  0x00 = 0b00000000 -> zero 1-bits (even)  -> parity bit = 1 (light OFF)

NOTE: Without the parity bit the RCX silently ignores all transmissions.
This is the single most common reason direct-to-RCX transmission fails.


RMT PULSE ENCODING
------------------
Consecutive bits of the same state are merged into a single longer pulse.

Example: three consecutive light-ON bits at 417us each = one 1251us pulse.

The ESP32 RMT peripheral receives a flat list of pulse durations. The
first pulse state (ON or OFF) is passed separately as the start value:
  start_val = 1 -> first pulse is a mark (light ON)
  start_val = 0 -> first pulse is a space (light OFF)

Since the first bit of every transmission is a START BIT (light ON),
start_val is always 1 for normal polarity.

Maximum pulse duration limit on ESP32 RMT: 32767 microseconds.
At 2400 baud the longest possible merged pulse is well within this limit.


EXAMPLE PAYLOAD
---------------
The following 9-byte sequence plays a sound on the RCX:

  0x55 0xFF 0x00 0x51 0xAE 0x05 0xFA 0x56 0xA9

  0x55 is the RCX sync/header byte. The RCX uses it to lock onto
  the baud rate and identify the start of a new message.


HARDWARE NOTES
--------------
- Use an NPN transistor (e.g. PN2222A) in common-emitter configuration
  to drive the IR LED with sufficient current for reliable transmission.
- GPIO HIGH -> transistor ON -> LED ON (normal polarity, matches protocol above).
- Base resistor: 330 ohm to 1k ohm. 10k ohm is too high and prevents
  the transistor from saturating, reducing LED drive current significantly.
- The RCX IR receiver is less sensitive than the LEGO IR tower.
  Transmit at close range (a few centimetres) aimed at the RCX IR window.
- The RCX tower has a more sensitive receiver than the RCX brick itself.
  Direct-to-RCX transmission requires more IR power than tower-to-RCX.


SUMMARY OF COMMON MISTAKES
---------------------------
1. Missing parity bit         -> RCX ignores all bytes silently
2. Wrong polarity             -> garbled signal
3. NEC or other IR protocol   -> completely wrong, RCX uses raw UART over IR
4. Base resistor too high     -> transistor does not saturate, weak IR signal
5. Wrong carrier duty cycle   -> RCX receiver may reject the signal
